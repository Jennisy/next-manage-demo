# 文章管理系统 — 后端知识学习指南

## 你能学到什么（后端视角）

这个项目不只是"写几个接口"，它覆盖了后端开发的核心知识体系。以下按模块拆解你将掌握的后端技能。

---

## 一、数据库设计与建模

### 1.1 关系型数据库核心概念

| 概念 | 你会学到的 | 实际场景 |
|------|-----------|---------|
| 主键（Primary Key） | 每条数据的唯一标识 | `Post.id` 自增主键 |
| 外键（Foreign Key） | 表与表之间的引用关系 | `Post.authorId` 引用 `User.id` |
| 联合主键 | 两个字段组合做唯一标识 | `PostTag(postId, tagId)` |
| 唯一约束（Unique） | 防止重复数据 | 分类名、标签名不能重复 |
| 索引（Index） | 加速查询 | 按 `authorId` 查文章时走索引 |
| 非空约束（NOT NULL） | 必填字段 | 文章标题不能为空 |
| 默认值（Default） | 不传时的兜底值 | `status` 默认 `draft` |

### 1.2 三大关联模式

```
一对多（1:N）
  User  ──→ Post     一个用户写多篇文章
  Category ──→ Post   一个分类下多篇文章
  实现方式：在"多"的那方加外键字段

多对多（M:N）
  Post ←──→ Tag      一篇文章多个标签，一个标签多篇文章
  实现方式：中间表 PostTag，两个外键组合

一对一（1:1）（本项目不涉及，但相关）
  User ──→ Profile   一个用户一份详细资料
  实现方式：在任一方加外键 + unique 约束
```

### 1.3 数据库迁移（Migration）

- **是什么**：用代码管理数据库结构的变更历史（类似 Git 管代码）
- **为什么**：多人协作时每个人的数据库结构必须一致，上线时不能手动改表
- **你会做的**：改 `schema.prisma` → `prisma migrate dev` → 自动生成 SQL 并执行

## 二、后端接口开发

### 2.1 RESTful API 设计原则

| 原则 | 说明 | 本项目示例 |
|------|------|-----------|
| 资源命名用名词复数 | URL 表示"资源集合" | `/api/posts`、`/api/categories` |
| HTTP 动词表示操作 | GET=查、POST=创建、PATCH=修改、DELETE=删除 | `PATCH /api/posts/1` |
| 状态码有意义 | 200 成功、400 参数错、401 未登录、403 无权限、404 不存在 | 删除不存在的文章返回 404 |
| 分页用 query 参数 | 不要把页码放在 URL 路径里 | `GET /api/posts?page=2&pageSize=10` |
| 筛选也用 query 参数 | 多条件筛选 | `GET /api/posts?categoryId=1&status=published` |

### 2.2 请求处理流程

一个接口从收到请求到返回响应，经历的完整链路：

```
请求进来
  │
  ├── 1. 中间件（Middleware）
  │     ├── 鉴权：token 是否有效？
  │     └── 权限：这个用户能访问这个接口吗？
  │
  ├── 2. 参数校验
  │     ├── 必填字段是否传了？
  │     ├── 类型对不对？（字符串/数字/枚举）
  │     └── 长度/范围是否合法？
  │
  ├── 3. 业务逻辑
  │     ├── 查数据库（读操作）
  │     ├── 写数据库（创建/更新/删除）
  │     └── 额外逻辑（检查关联数据是否存在等）
  │
  ├── 4. 错误处理
  │     ├── 数据库连接失败 → 500
  │     ├── 数据不存在 → 404
  │     └── 唯一约束冲突 → 400（"分类名已存在"）
  │
  └── 5. 统一响应格式
        └── { code, data, message }
```

### 2.3 你会写的接口类型

| 类型 | 复杂度 | 代表接口 |
|------|--------|---------|
| 简单 CRUD | ⭐ | 分类/标签的增删改查 |
| 带关联的创建 | ⭐⭐ | 创建文章时同时关联标签 |
| 复合条件查询 | ⭐⭐⭐ | 文章列表（分页 + 多条件筛选 + 模糊搜索） |
| 聚合统计 | ⭐⭐⭐ | 每人发文数、每分类文章数 |
| 事务操作 | ⭐⭐⭐ | 更新文章标签（先删旧的再加新的，必须都成功） |

## 三、ORM 与数据操作

### 3.1 Prisma 核心操作速查

| 操作 | 方法 | 等价 SQL |
|------|------|---------|
| 查单条 | `findUnique({ where: { id } })` | `SELECT * WHERE id = ?` |
| 查列表 | `findMany({ where, orderBy })` | `SELECT * WHERE ... ORDER BY ...` |
| 创建 | `create({ data })` | `INSERT INTO ...` |
| 更新 | `update({ where, data })` | `UPDATE ... SET ... WHERE ...` |
| 删除 | `delete({ where })` | `DELETE FROM ... WHERE ...` |
| 计数 | `count({ where })` | `SELECT COUNT(*)` |
| 分组统计 | `groupBy({ by, _count })` | `GROUP BY ... COUNT(*)` |

### 3.2 关联操作（本项目重点）

```ts
// 创建文章时关联已有标签
prisma.post.create({
  data: {
    title: '标题',
    content: '内容',
    author: { connect: { id: 1 } },        // 关联已有用户
    category: { connect: { id: 2 } },      // 关联已有分类
    tags: {
      create: [                             // 创建中间表记录
        { tag: { connect: { id: 1 } } },
        { tag: { connect: { id: 3 } } }
      ]
    }
  }
})

// 更新文章标签（先删后建）
prisma.$transaction([
  prisma.postTag.deleteMany({ where: { postId: 1 } }),
  prisma.postTag.createMany({ data: [
    { postId: 1, tagId: 2 },
    { postId: 1, tagId: 5 }
  ]})
])
```

### 3.3 事务（Transaction）

**什么时候需要事务**：一组操作必须全部成功或全部失败。

| 场景 | 为什么需要事务 |
|------|--------------|
| 更新文章标签 | 删除旧关联 + 创建新关联，中间失败了数据就乱了 |
| 删除分类并迁移文章 | 先把文章移到其他分类，再删除分类 |
| 批量导入文章 | 100 篇文章要么全部导入成功，要么一篇都不入库 |

## 四、权限与安全

### 4.1 认证（Authentication） vs 授权（Authorization）

| | 认证 | 授权 |
|---|---|---|
| 回答的问题 | "你是谁？" | "你能干什么？" |
| 实现方式 | JWT token 验证 | 角色判断（管理员/编辑/访客） |
| 失败返回 | 401 Unauthorized | 403 Forbidden |
| 本项目场景 | 登录后才能访问系统 | 只有管理员能删除别人的文章 |

### 4.2 接口级权限控制

```ts
// 你会实现这种逻辑：
// 删除文章 —— 管理员可以删任何人的，普通用户只能删自己的
if (currentUser.role !== '管理员' && post.authorId !== currentUser.id) {
  return fail('无权删除他人文章', 403)
}
```

### 4.3 数据校验

- **前端校验**：用户体验（即时反馈），可以被绕过
- **后端校验**：安全保障（最终防线），不可或缺

你会学到：即使前端做了校验，后端也**必须**再校验一次。因为接口可以被 Postman/curl 直接调用，绕过前端。

## 五、错误处理

### 5.1 错误分层

| 层级 | 错误类型 | 处理方式 |
|------|---------|---------|
| 参数层 | 缺少必填字段、类型错误 | 400 + 明确提示 |
| 业务层 | 分类名重复、文章不存在 | 400/404 + 业务提示 |
| 数据库层 | 唯一约束冲突、外键约束失败 | 捕获特定错误码，转为业务提示 |
| 系统层 | 数据库连接断开、未知异常 | 500 + 通用错误（不暴露内部细节） |

### 5.2 Prisma 常见错误码

| 错误码 | 含义 | 你的处理 |
|--------|------|---------|
| P2002 | 唯一约束冲突 | "该分类名已存在" |
| P2003 | 外键约束失败 | "关联的分类/用户不存在" |
| P2025 | 记录不存在 | "文章不存在或已被删除" |

## 六、性能意识

### 6.1 N+1 问题

```ts
// ❌ 错误写法：查 10 篇文章，每篇再单独查作者 = 11 次数据库请求
const posts = await prisma.post.findMany()
for (const post of posts) {
  post.author = await prisma.user.findUnique({ where: { id: post.authorId } })
}

// ✅ 正确写法：一次查询带出关联数据 = 1 次数据库请求
const posts = await prisma.post.findMany({
  include: { author: true }
})
```

### 6.2 只查需要的字段

```ts
// ❌ 文章列表不需要正文内容，但 include 全部字段了
prisma.post.findMany({ include: { author: true } })

// ✅ 列表只要摘要，详情才要正文
prisma.post.findMany({
  select: {
    id: true, title: true, summary: true, createdAt: true,
    author: { select: { id: true, name: true } }
  }
})
```

### 6.3 分页的必要性

- 数据量小时感觉不到差异
- 数据量到 1000+ 时，不分页 = 一次查询返回所有数据 = 慢 + 占内存
- 后端只返回当前页的数据 + 总条数，前端按需请求

## 七、学完后你具备的后端能力

| 能力 | 具体表现 |
|------|---------|
| 独立设计数据表 | 给你一个业务需求，你能画出 ER 图确定表结构和关联 |
| 独立写 CRUD 接口 | 包括参数校验、关联操作、错误处理、统一响应 |
| 复杂查询 | 多条件筛选、模糊搜索、分页、聚合统计 |
| 权限控制 | 接口级别的角色权限判断 |
| 事务处理 | 多步操作保证数据一致性 |
| 性能优化意识 | 避免 N+1、按需查询字段、合理使用索引 |

## 八、跟前端学习指南的对比

| 维度 | 前端指南侧重 | 后端指南侧重 |
|------|-------------|-------------|
| 数据 | 怎么用、怎么渲染 | 怎么存、怎么查、怎么保护 |
| 接口 | 怎么调、参数怎么传 | 怎么设计、怎么校验、怎么处理错误 |
| 性能 | 减少请求次数、缓存 | 减少数据库查询次数、索引优化 |
| 安全 | 防 XSS、token 管理 | 参数校验、权限控制、SQL 注入防护 |

两份指南互补，前端指南告诉你"为什么接口长这样"，后端指南告诉你"怎么把接口写出来"。

## 九、推荐学习顺序

```
第 1 步：Schema 设计 + Migration
         → 理解"数据怎么存"

第 2 步：分类/标签简单 CRUD
         → 热身，跑通完整链路

第 3 步：文章 CRUD（含关联）
         → 核心，练一对多 + 多对多

第 4 步：复合查询 + 分页
         → 后端最有价值的技能

第 5 步：统计接口
         → groupBy 聚合思维

第 6 步：权限细化
         → 管理员 vs 普通用户的差异处理
```
