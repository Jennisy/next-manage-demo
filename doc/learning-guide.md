# 文章管理系统 — 前端视角学习指南

## 你能学到什么

作为前端开发者，通过这个项目你将补齐「数据层」的知识短板，从"只会调接口"进阶到"能设计数据模型、写后端逻辑"。

---

## 一、核心技能收获

### 1. 数据库表设计思维

**你目前的认知**：前端拿到 JSON 直接渲染。

**学完后的认知**：理解数据从哪来、怎么存、为什么接口返回的结构是这样。

| 学习点 | 对应场景 | 前端价值 |
|--------|---------|---------|
| 主键/外键 | Post.authorId → User.id | 理解为什么详情接口能"带出"作者信息 |
| 一对多 | 一个用户有多篇文章 | 理解列表接口里嵌套对象的来源 |
| 多对多 + 中间表 | 文章 ↔ 标签 | 理解为什么标签要单独一个接口，而不是存在文章字段里 |
| 唯一约束 | 分类名不能重复 | 理解后端返回"已存在"错误的底层原因 |
| 级联删除 | 删文章自动清标签关联 | 理解为什么有些删除需要确认、有些直接删 |

---

### 2. SQL/ORM 查询能力（通过 Prisma）

你不需要手写 SQL，Prisma 提供了 TypeScript 友好的 API，但学到的**查询思维**跟 SQL 是通用的：

| 查询模式 | 做什么 | 等价 SQL 概念 |
|---------|--------|--------------|
| `include` | 查文章时一起带出作者、标签 | JOIN |
| `where` + 多条件 | 按分类 + 关键词 + 状态筛选 | WHERE AND |
| `contains` | 标题模糊搜索 | LIKE '%keyword%' |
| `skip` / `take` | 分页 | LIMIT OFFSET |
| `orderBy` | 按时间排序 | ORDER BY |
| `groupBy` + `_count` | 统计每人发文数 | GROUP BY + COUNT |
| `some` | 查含有某标签的文章 | EXISTS 子查询 |
| 事务（transaction） | 创建文章 + 关联标签一起成功或失败 | BEGIN...COMMIT |

**这些是通用数据库知识**，以后用 MySQL、MongoDB、甚至 GraphQL 都是同一套思路。

### 3. 接口设计能力

| 学习点 | 具体收获 |
|--------|---------|
| RESTful 设计 | 什么时候用 GET/POST/PATCH/DELETE，URL 怎么组织 |
| 分页协议 | 前后端如何约定分页参数和返回格式 |
| 筛选参数设计 | query string 怎么传多条件，后端怎么解析 |
| 错误处理 | 删除有文章的分类 → 应该返回什么错误 |
| 权限粒度 | "管理员能删所有文章，普通用户只能删自己的"怎么实现 |

学完这些，以后跟后端对接口时你能直接参与设计讨论，而不是被动等文档。

### 4. 前端进阶技能（附带收获）

| 场景 | 技能 |
|------|------|
| 文章列表多条件筛选 | URL query 状态管理、防抖搜索 |

## 二、学习路径建议

按这个顺序学，每步都能跑起来看到效果：

```
第一周：数据模型 + 基础 CRUD
├── Day 1-2：设计 Schema、跑 migrate、写 seed 数据
├── Day 3-4：分类 + 标签的 CRUD 接口（最简单的热身）
└── Day 5-7：文章 CRUD 接口（核心，一对多 + 多对多）

第二周：复杂查询 + 前端页面
├── Day 1-2：文章列表接口（分页 + 筛选 + 搜索）
├── Day 3-4：统计接口（groupBy 聚合）
└── Day 5-7：前端页面（列表、表单、详情）
```

## 四、重点难点预警

| 难点 | 为什么难 | 建议 |
|------|---------|------|
| 多对多中间表 | 概念不直觉——为什么不能直接存数组 | 想象成"一张登记表"，记录谁和谁有关系 |
| 创建文章时关联标签 | 要同时操作两张表 | 用 Prisma 的 `create` + `connect` 语法，一条语句搞定 |
| 分页 + 筛选组合 | 条件越多 where 越复杂 | 先写死一个条件跑通，再逐个加 |
| groupBy 聚合 | 概念抽象 | 类比 JS 的 `reduce` 分组统计 |

## 五、需求文档没问题

看了需求文档，设计合理，几点确认：

- ✅ 表结构覆盖一对多、多对多两种核心关联
- ✅ 接口设计遵循你已有的 `apiResponse` 规范
- ✅ 查询场景丰富（筛选、搜索、分页、统计都有）
- ✅ 实施步骤由简到难，循序渐进
- ✅ 前端页面有东西可做（不是纯后端练习）

没有需要修改的地方，可以按步骤开始实施。
